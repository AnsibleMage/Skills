# 프롬프트 가이드 — Wireframe Skill

## 기본 문법

```
/wireframe [feature-description]
```

---

## 1. 자동 감지 키워드

### 1.1 최적화 의도 (포함 시 목표 질문 스킵)

| 키워드 패턴 | 감지되는 의도 |
|------------|-------------|
| `simplify`, `reduce clutter` | 단순화 |
| `more compact`, `more consistent` | 밀도/일관성 개선 |
| `make it more [형용사]` | 방향성 개선 |
| `improve [측면]` | 특정 측면 개선 |
| `redesign for [목표]` | 목표 지향 재설계 |
| `modernize` | 현대화 |

### 1.2 스코프 (포함 시 Phase 2 질문 스킵)

| 키워드 | 효과 |
|--------|------|
| `wireframe and visuals` | Phase 2 (Clean + Polished) 자동 실행 |
| `wireframe and color variants` | 동일 |
| `create wireframes and visuals` | 동일 |
| (미언급) | "Wireframes only vs +visuals?" 질문 발생 |

---

## 2. 질문 흐름 제어

### 2.1 최대 질문 수

| 상황 | 질문 수 |
|------|--------|
| 첫 실행 (design-context 없음) | 스크린샷 요청 + 플랫폼 질문 + 최적화 + 스코프 = 최대 4개 |
| 재실행 + 중립 기능명 | 최적화 + 스코프 = 최대 2개 |
| 재실행 + 의도 + visuals 명시 | 0개 (바로 생성) |

### 2.2 질문 제로 공식

```
/wireframe [and visuals for] [기능] [with/for 최적화-의도]
```

---

## 3. 실전 프롬프트 예시

### 3.1 탐색적 (질문 있음)

```
/wireframe article page
/wireframe settings dashboard
/wireframe user profile
```
> 최적화 목표 질문 O, 스코프 질문 O

### 3.2 방향 제시 (질문 1개)

```
/wireframe simplify the signup flow
/wireframe make the dashboard more compact
/wireframe redesign header for better discoverability
/wireframe reduce clutter on settings page
/wireframe modernize the checkout experience
```
> 최적화 자동 감지, 스코프 질문만 O

### 3.3 풀옵션 (질문 0개)

```
/wireframe wireframe and visuals for dashboard with better data density
/wireframe wireframe and visuals for signup flow simplify
/wireframe wireframe and color variants for landing page more conversions
```
> 바로 Phase 1 + Phase 2 실행

### 3.4 B&W만 (Phase 1 only)

```
/wireframe checkout flow
→ 스코프 질문에서 "Wireframes only" 선택
```

## 4. 최적화 목표 옵션

스킬이 제시하는 4가지 사전 정의 목표:

| 목표 | 효과 |
|------|------|
| **More conversions** | CTA 우선, 마찰 감소, 설득적 레이아웃 |
| **Less drop-offs** | 단순화, 진행 표시, 오류 방지 |
| **More time on page** | 콘텐츠 깊이, 탐색 경로, 참여 유도 |
| **Better discoverability** | 탐색, 검색, 정보 향기(information scent) |

> 자유 텍스트도 가능 — 위 4개에 국한되지 않음

---

## 5. 출력 구조

```
wireframe/DDMM-feature-name/
├── index.html          ← 브라우저에서 열기
├── base.css            ← 프레임워크 (수정 불필요)
├── styles.css          ← B&W 와이어프레임
├── styles-opt1~5.css   ← Clean + Polished 변형 (Phase 2)
```

- **5개 옵션**: Option 1 (Safe) + Options 2~5 (Exploratory)
- **3개 서브탭/옵션**: Wireframe (B&W) / Clean (플랫 컬러) / Polished (그래디언트+애니메이션)
- **Summary 탭**: 스코어링 테이블 + 추천

---

## 6. 첫 실행 vs 재실행

| 항목 | 첫 실행 | 재실행 |
|------|--------|--------|
| 코드베이스 스캔 | O | X (캐시됨) |
| 스크린샷 요청 | O (2~3장) | X |
| 플랫폼 질문 | O (Mobile/Desktop/Both) | X |
| design-context.md | 생성 | 읽기만 |
| design-taste.md | 복사 | 읽기만 |

> 첫 실행에서 `wireframe/brain/design-context.md`가 생성되면 이후 모든 실행에서 재사용

---

## 7. 팁

- **컴포넌트만 와이어프레임**: 전체 페이지가 아닌 단일 컴포넌트도 가능 (예: `/wireframe search bar component`)
- **기존 패턴 유지**: Option 1 (Safe)은 항상 현재 디자인 시스템을 확장 — 비교 기준선으로 활용
- **반복 실행**: 같은 프로젝트에서 다른 기능을 여러 번 실행하면 `wireframe/` 아래 날짜별 폴더로 누적
- **design-context 갱신**: 새 패턴 발견 시 자동으로 `design-context.md`에 추가됨
